Patrones de Diseño

Patrones implementados en la aplicación

Service Layer

Capa de servicios como intermediaria entre controladores y la API externa. Cada módulo tiene su servicio dedicado con operaciones CRUD estandarizadas.

// Flujo de cada operación
Controller
  → Service
    → ApiRequests
      → SendHttpClient
        → REST API :7076

Facade

ApiRequests actúa como fachada que abstrae la complejidad de SendHttpClient, proporcionando una interfaz simplificada a los servicios.

internal static class ApiRequests
{
  ListAsync<T>(url);
  FindAsync<T>(url, id);
  CreateAsync(url, model);
  EditAsync(url, model);
  DeleteAsync(url, id);
}

DTO (Data Transfer Object)

Objetos de transferencia de datos especializados por operación. Separan el modelo de dominio del modelo de transporte.

ConceptoPagoListDto → Listados
ConceptoPagoDetailDto → Detalle
ConceptoPagoFindDto → Búsqueda
ConceptoPagoDropdownDto → Selects

ViewModel

ViewModels con Data Annotations para validación. Variantes por tipo de vista y operación.

CargoViewModel → Create/Edit
CargoDetailViewModel → Detail
AlumnoListViewModel → DataTable
DropdownViewModel → Selects

Module Pattern (JS)

Encapsulación frontend con IIFE y namespacing. Módulos como Modals y DataTable evitan contaminación del scope global.

Async/Await (TAP)

Programación asíncrona basada en Task en toda la arquitectura: controladores, servicios y comunicación HTTP.

{

  Task<bool> CreateAsync(url, model);
  Task<bool> EditAsync(url, model);

Flujo de Autenticación

1. Login form → AccountController
2. AccountService → POST /api/v1/Account/Login
3. API returns JWT + User data
4. Session["Token"] = JWT
5. Session["UserId"], ["Username"], ["RoleName"]
6. Redirect → Home/Index
